System Overview

This is a Hardware in the Loop (HIL) control systems project from MECH471 at Concordia. A controller running on an Arduino Uno drives a virtual RC car in real time: a second Arduino runs a vehicle and tyre model as the plant, and the two boards talk over plain GPIO. The controller emits two servo style pulse trains (drive and steering) and reads three analog wheel speed signals back from the plant, closing the loop every cycle.

I built the controller side of this system from the registers up: the low level AVR firmware (timers, ADC, digital I/O, and a reusable PID library) and the PlatformIO build system that ties the controller and simulator targets together. The vehicle and tyre simulation that the controller runs against was provided by the course as the plant to control.

The controller and plant exchange the following signals:

  • u1, drive motor command, sent as a 1000 to 2000 microsecond servo pulse on D7
  • u2, steering command, sent as a servo pulse on D8
  • y1, rear wheel speed, read on analog input A1
  • y2, right front wheel speed, read on A3
  • y3, left front wheel speed, read on A5
  • A shared ground between the two boards

Low Level Firmware (custom AVR HAL)

Rather than lean on the Arduino library, I wrote my own hardware abstraction layer for the ATmega328P so I had direct control over timing and I/O. It is split into small modules: a Timer2 based millisecond clock with a measured loop time step for the controllers, buffered multi channel ADC sampling for the three wheel speed inputs, a thin digital I/O layer, and a reusable PID library.

Timing & servo PWM

The two servo pulses are generated directly from Timer1. A timer overflow starts both pulses and flags the main loop to run, and two output compare interrupts end each pulse at the width that encodes the current command, so pulse timing is handled in hardware and stays jitter free regardless of what the main loop is doing. 

ISR(TIMER1_OVF_vect) {
    PORTD |=  BIT(PORTD7);   // D7 HIGH: start u1 pulse (drive motor)
    PORTB |=  BIT(PORTB0);   // D8 HIGH: start u2 pulse (steering)
    control_flag = true;     // flag the main loop to run
}

ISR(TIMER1_COMPA_vect) {
    PORTD &= ~BIT(PORTD7);   // D7 LOW: end u1 pulse
}

ISR(TIMER1_COMPB_vect) {
    PORTB &= ~BIT(PORTB0);   // D8 LOW: end u2 pulse
}

A small mapping function turns a commanded armature voltage into a pulse width, centred on the 1500 microsecond neutral point and clamped to the safe servo range so a bad command can never drive the output out of bounds: 

uint16_t va_to_pw(float Va) {
    float pw = (Va / V_BAT) * PW_RANGE + PW_NEUTRAL;
    if (pw < PW_MIN) pw = PW_MIN;
    if (pw > PW_MAX) pw = PW_MAX;
    return (uint16_t)pw;
}

ADC driver (register level)

The wheel speed signals arrive as analog voltages, so I wrote the ADC driver straight against the registers. ADC_INIT() enables the converter and sets the /128 prescaler so the ADC clock lands at 125 kHz (16 MHz divided by 128), which keeps it inside the 50 to 200 kHz range the hardware needs for a full 10 bit conversion. It also right adjusts the result and selects AVcc as the reference. A blocking read then selects the channel through the low nibble of ADMUX, starts a conversion, polls until it finishes, reads the 10 bit result, and scales it to volts. 

void ADC_INIT(void){
    ADCSRA |= BIT(ADEN);   // enable ADC
    // prescaler /128: 16 MHz / 128 = 125 kHz ADC clock (within the 50 to 200 kHz spec)
    ADCSRA |= BIT(ADPS0);
    ADCSRA |= BIT(ADPS1);
    ADCSRA |= BIT(ADPS2);
    ADMUX  &= ~BIT(ADLAR); // right adjust the result
    ADMUX  |= BIT(REFS0);  // AVcc reference
    ADMUX  &= ~BIT(REFS1);
}

float read_analog_ADC(ANALOG_PINS pin){
    if (pin > 6)
        return 0.0;                          // reject an invalid channel
    ADMUX = ((ADMUX & 0xF0) | (uint8_t)pin); // select the channel
    ADCSRA |= BIT(ADSC);                     // start conversion
    while (ADCSRA & BIT(ADSC));              // wait until done
    uint16_t result = ADC;                   // read ADCL then ADCH
    return result * (5.0/1023.0);            // scale to volts
}

Interrupt driven buffered ADC

Reading three wheel speeds with the blocking call above would stall the control loop, so the running system uses an interrupt driven scanner built on top of that driver. Compile time flags pick which channels are active, and each one gets its own FIFO ring buffer. init_buffer() points the ADC at the first channel, enables the conversion complete interrupt, and starts the first conversion. From then on the ADC free runs: each time a conversion completes, the interrupt pushes the sample into that channel's buffer, advances to the next enabled channel in a round robin, repoints the multiplexer, and kicks off the next conversion. The main loop never waits on the ADC; it just reads a boxcar average out of each buffer, which also smooths the noisy wheel speed signal. 

// abridged: one case per enabled channel
ISR(ADC_vect){
    uint16_t val = ADC;
    switch (enabled_pins[current_index]){
        case A_1:
            LIB_BUFFER_FIFO_INSERT(&USE_Buffer1, val);
            break;
        // ... one case per enabled channel ...
    }
    current_index++;
    if (current_index >= NUM_PINS)
        current_index = 0;
    uint8_t next_pin = enabled_pins[current_index];
    ADMUX = ((ADMUX & 0xF0) | next_pin); // point at the next channel
    ADCSRA |= BIT(ADSC);                  // start its conversion
}

uint16_t buffer_avg(uint16_t *arr){
    uint32_t sum = 0;
    for (int i = 0; i < BufferLength; i++)
        sum += arr[i];
    return sum / BufferLength;
}

PID library

The PID library is my own as well, and it handles the details that make a discrete PID behave on real hardware. It uses the actual measured time step each cycle rather than an assumed constant, with a guard against division by zero. It takes the derivative on the measurement instead of the error, so a step change in setpoint does not produce a derivative kick, and it low pass filters that derivative through an alpha term to reject sensor noise. The integrator is clamped between fixed bounds for anti windup, so it cannot run away while the output is saturated. 

float PID_compute(PID_t *pid, float setpoint, float measurement, struct time_differencePID *time){
    float error = setpoint - measurement;
    update_dt(time);
    float dt = time->dt / 1000.0f;
    if (dt <= 0.0f) dt = 1e-6f;                 // guard against divide by zero

    pid->proportional = pid->kp * error;

    pid->integrated_error += error * dt;        // integrate, then clamp for anti windup
    if (pid->integrated_error > pid->integral_max)
        pid->integrated_error = pid->integral_max;
    else if (pid->integrated_error < pid->integral_min)
        pid->integrated_error = pid->integral_min;
    pid->integral = pid->integrated_error * pid->ki;

    // derivative on the measurement (not the error) avoids derivative kick
    pid->derivative = ((measurement - pid->prev_measurement) / dt) * pid->kd;
    pid->prev_measurement = measurement;
    // first order low pass filter on the derivative term
    pid->derivative = pid->alpha * pid->derivative + (1 - pid->alpha) * pid->prev_derivative;
    pid->prev_derivative = pid->derivative;

    return (pid->proportional + pid->integral + pid->derivative);
}

Control Strategy

On top of that firmware, the controller runs four control modes at 50 Hz, arbitrated by a priority stack that picks the right mode for the vehicle state each cycle:

  • Cruise / speed control, a PID loop that holds a target wheel speed
  • Launch control, an open loop voltage applied from a standstill until the car reaches a speed threshold, which keeps the speed PID from winding up at zero speed before handing over to it
  • Traction control, which watches the slip ratio between the rear and front wheels and backs off throttle when the rear spins up, and loosens its slip margin while steering
  • Braking control, a separate loop that manages slip during deceleration

Slip is computed from the front and rear wheel speeds, with a low speed cutoff so noise at a near standstill cannot produce a meaningless ratio, and a saturation so a single bad sample cannot dominate the controller: 

float compute_slip(float w_front, float w_rear) {
    const float MIN_FRONT_SPEED = 30.0f;

    if (fabsf(w_front) < MIN_FRONT_SPEED) {
        return 0.0f;
    }

    float S = (w_front - w_rear) / fabsf(w_front);

    if (S >  3.0f) S =  3.0f;
    if (S < -3.0f) S = -3.0f;

    return S;
}

A maneuver sequencer drives the whole thing through a scripted routine of forward and reverse runs, turns, and a donut, which exercises each control mode in turn. Every cycle is logged as CSV over serial at up to 1 Mbaud so the runs can be plotted and the controllers tuned against real traces.

Build System & Simulation

I chose PlatformIO over the Arduino IDE so the project could hold several firmware images in one repository and build them reproducibly from the command line. A single platformio.ini defines four environments that share the same board and framework but compile different sources through build source filters: the controller, and the three plant simulators of increasing fidelity. The same setup runs as a two board simulation in Wokwi, so the full HIL loop can be exercised in the browser without hardware. 

[platformio]
default_envs = uno

[env:uno]                       ; controller
platform = atmelavr
board = uno
framework = arduino
build_src_filter = -<*> +<main.cpp> +<HW/>
monitor_speed = 1000000

[env:simulator3]                ; provided vehicle + tyre plant
platform = atmelavr
board = uno
framework = arduino
build_src_filter = -<*> +<simulator/sim3/HIL3prof.cpp>
monitor_speed = 2000000

Switching what gets flashed is then a one line change of environment, and the controller and plant can be uploaded to their two boards independently.
